# Interactive Server API Docs

For LoRACK server APIs, the best approach is to use OpenAPI as the source of truth and render it inside Docusaurus.

Recommended stack:

- `docusaurus-plugin-openapi-docs`
- `docusaurus-theme-openapi-docs`

This gives you generated endpoint pages, request/response schemas, auth documentation, and interactive request support in one docs site.

## 1) Add dependencies

From `docs-site/`:

```bash
npm install @docusaurus/core@^3.10.0 @docusaurus/preset-classic@^3.10.0 @docusaurus/faster@^3.10.0
npm install docusaurus-plugin-openapi-docs@^5.0.0 docusaurus-theme-openapi-docs@^5.0.0 docusaurus-plugin-sass@^0.2.6 sass
```

## 2) Add an OpenAPI spec

Create a spec file in the server workspace, for example:

- `server/openapi/lorack-services.yaml`

At minimum, include these tags and paths:

- `Auth`: `POST /api/v1/users/login`
- `Users`: `GET /api/v1/users/me`, `GET /api/v1/users`, `POST /api/v1/users`, `PUT /api/v1/users/{id}`
- `History`: health scores and battery readings endpoints
- `Contacts`: sync/export/import endpoints
- `AI`: snapshots/insights/model endpoints
- `Fleet`: snapshot/dashboard/coverage/alerts endpoints
- `Device Profiles`: popular/search/detail/contribute endpoints
- `Health`: `GET /health`

Define both auth schemes because LoRACK supports either one:

- Bearer JWT (`Authorization: Bearer ...`)
- API Key (`X-API-Key: user_...`)

## 3) Wire plugin and theme in Docusaurus

Update `docs-site/docusaurus.config.ts`:

```ts
import { themes as prismThemes } from 'prism-react-renderer';
import type { Config } from '@docusaurus/types';
import type * as Preset from '@docusaurus/preset-classic';

const config: Config = {
  // ...existing config...

  themes: ['docusaurus-theme-openapi-docs'],

  plugins: [
    [
      'docusaurus-plugin-openapi-docs',
      {
        id: 'openapi',
        docsPluginId: 'classic',
        config: {
          lorackServices: {
            specPath: '../server/openapi/lorack-services.yaml',
            outputDir: 'docs/api/server',
            sidebarOptions: {
              groupPathsBy: 'tag',
              categoryLinkSource: 'tag',
            },
          },
        },
      },
    ],
  ],

  presets: [
    [
      'classic',
      {
        docs: {
          // keep your existing options
          sidebarPath: './sidebars.ts',
          routeBasePath: '/',
          docItemComponent: '@theme/ApiItem',
        },
      } satisfies Preset.Options,
    ],
  ],
};

export default config;
```

## 4) Add scripts for API doc generation

Update `docs-site/package.json` scripts:

```json
{
  "scripts": {
    "gen-api-docs": "docusaurus gen-api-docs lorackServices --plugin-id openapi",
    "clean-api-docs": "docusaurus clean-api-docs lorackServices --plugin-id openapi"
  }
}
```

Then generate pages:

```bash
npm run gen-api-docs
```

## 5) Add API section to sidebar

Update `docs-site/sidebars.ts` to include generated API docs:

```ts
import type { SidebarsConfig } from '@docusaurus/plugin-content-docs';

const sidebars: SidebarsConfig = {
  docsSidebar: [
    { type: 'doc', id: 'index', label: 'Home' },
    { type: 'doc', id: 'user-guide/index', label: 'User Guide' },
    {
      type: 'category',
      label: 'API',
      items: [{ type: 'autogenerated', dirName: 'api/server' }],
    },
  ],
};

export default sidebars;
```

## 6) Verify locally

From `docs-site/`:

```bash
npm run start
```

From `server/` (for live Try-It requests):

```bash
npm run dev
```

If Try-It calls fail in browser, verify `CORS_ORIGINS` includes your docs site origin.

## Notes for LoRACK

- Keep endpoint descriptions focused on mobile usage and auth mode selection (`X-API-Key` vs Bearer).
- Keep examples for resource-scoped routes (`resourceType` + `resourceId`) and profile contacts (`resourceType=profile`).
- Treat OpenAPI spec updates as part of API route changes so docs stay in sync.
